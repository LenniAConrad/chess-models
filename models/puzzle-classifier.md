# Puzzle Classifier

`puzzle-classifier_21planes-6blocksx64-head32-logit1.bin` is a binary chess
position classifier trained from ChessRTK stack analysis records.

The model answers one question:

```text
Does this position look like a puzzle candidate?
```

The output is a single logit. Higher logits mean the positive class is more
likely.

## Architecture

- Input: `21 x 8 x 8`
- Stem: `3x3` convolution, `21 -> 64`
- Trunk: `6` residual blocks, `64` channels
- Head: `1x1` convolution, `64 -> 32`
- Pooling: global average pool
- Output: dense `32 -> 1`

The exported `.bin` folds batch-normalization parameters into convolution
weights and biases, which is the layout loaded by ChessRTK's
`chess.nn.classifier.Network`.

Serialized parameter count:

```text
457,409
```

## Input Planes

The input is encoded from the side-to-move perspective:

1. our pawns
2. our knights
3. our bishops
4. our rooks
5. our queens
6. our king
7. opponent pawns
8. opponent knights
9. opponent bishops
10. opponent rooks
11. opponent queens
12. opponent king
13. repetition
14. our queen-side castling right
15. our king-side castling right
16. opponent queen-side castling right
17. opponent king-side castling right
18. side-to-move-is-black
19. halfmove clock
20. unused zero plane
21. edge plane

## Dataset

The training dataset is exported from the USB stack directories:

```text
/media/lennart/USB STICK/1M_STACK
/media/lennart/USB STICK/2M_STACK
```

The export command is generated by:

```bash
training/classifier/run_full_usb_pipeline.sh
```

The pipeline writes local data to:

```text
training/classifier/data/usb-stacks-puzzle-focused/
```

Those files are intentionally ignored by git because they can be large:

```text
*.classifier.inputs.npy
*.classifier.labels.npy
*.classifier.meta.json
```

Labels are binary:

- `1.0`: puzzle-like
- `0.0`: non-puzzle-like

ChessRTK's `record-to-classifier` exporter uses explicit `kind=puzzle` labels
when present, otherwise the configured puzzle verification fallback.

## Training Pipeline

Run the USB export, train with early stopping, and export the best
checkpoint to the model file:

```bash
training/classifier/run_full_usb_pipeline.sh
```

Useful overrides:

```bash
EPOCHS=120 PATIENCE=8 BATCH_SIZE=2048 LR=0.0003 training/classifier/run_full_usb_pipeline.sh
```

The pipeline scans every USB stack file but writes a puzzle-focused tensor
dataset:

```text
training/classifier/data/usb-stacks-puzzle-focused/
```

That focused dataset contains every real puzzle row discovered during the full
USB scan and a capped number of real non-puzzle rows. The default negative cap
is:

```text
MAX_NEGATIVES=3000000
```

With the `21 * 64` float32 input format, this leaves room for roughly 6.5
million actual puzzle rows while keeping the focused tensors around the 50 GiB
budget. The trainer uses memory-mapped NumPy arrays, so it does not load the
whole dataset into RAM.

CUDA is selected automatically when PyTorch can see a CUDA device. Device
selection can also be forced:

```bash
DEVICE=auto training/classifier/run_full_usb_pipeline.sh
DEVICE=cuda CUDA_DEVICE=0 training/classifier/run_full_usb_pipeline.sh
DEVICE=cpu training/classifier/run_full_usb_pipeline.sh
```

Run metadata records the selected device, CUDA availability, GPU name, compute
capability, and GPU memory when CUDA is used.

Set `MAX_NEGATIVES` higher or lower depending on the actual number of puzzles
and available memory. Set `WEIGHTED_LOSS=1` to additionally enable weighted
binary cross-entropy.

The pipeline automatically writes:

```text
models/puzzle-classifier_21planes-6blocksx64-head32-logit1.bin
```

Each run directory under `training/classifier/runs/` contains:

- `model.pt`: best PyTorch checkpoint
- `metrics.json`: training history and held-out metrics
- `export.sha256`: SHA-256 for the exported `.bin`
- `exported-model.txt`: exported model path

Run outputs are ignored by git.

## Evaluation

For imbalanced puzzle classification, accuracy must be compared with the
all-negative baseline. If the puzzle rate is `20%`, an always-negative model
gets `80%` accuracy.

Primary metrics:

- ROC-AUC
- average precision
- top-k precision at the number of true positives
- recall at the selected operating threshold

Accuracy is still reported, but it is not sufficient by itself.
